# Webcam 项目原理说明

## 1. 这个项目本质上在做什么

这个 `webcam_app` 项目的目标，不是重新训练模型，而是把**已经训练好的模型**接到**实时视频流**上，让模型在电脑上持续处理摄像头画面，并把结果显示、保存出来。

它当前主要服务一个方向：

1. **YOLO 实时检测**
   适合自动判断画面里有没有目标，并给出检测框。

所以这个项目本质上是一个“**推理端工程**”，不是“训练端工程”。

---

## 2. 整体工作原理

整个项目的核心链路可以概括成一句话：

**摄像头采集画面 -> 送进模型推理 -> 把结果重新画回图像 -> 在窗口显示/保存**

如果展开成流程图，就是：

```text
摄像头 / 视频源
      ↓
OpenCV 读取每一帧图像
      ↓
模型推理（YOLO）
      ↓
后处理（画框 / 统计信息）
      ↓
窗口实时显示
      ↓
可选保存到视频、标签、日志
```

这个流程里，最关键的是“**逐帧处理**”。

也就是说，程序不是一次性把整段视频全读完再算，而是：

- 读 1 帧
- 推理 1 帧
- 显示 1 帧
- 再继续下一帧

这就是它能够做“实时检测”的原因。

---

## 3. 当前项目里接的是谁

### 3.1 YOLO 路线

如果你运行的是：

```powershell
python scripts/run_webcam.py
```

或者：

```powershell
python scripts/run_webcam.py --source 0 --backend dshow
```

那么走的是 **YOLO 路线**。

当前默认权重来自：

```text
../yolo11n_train_v3_datachanged_2/weights/best.pt
```

也就是说，这条路线接的是**你自己训练出来的模型**。

---

## 4. 项目为什么拆成多个脚本

这个项目不是把所有功能都塞进一个大文件里，而是按职责拆开了。

### `common.py`

这是公共工具模块，负责：

- 统一路径管理
- 输出目录管理
- 摄像头后端映射
- 安全文件名处理

它的作用是让别的脚本不用重复写同样的路径和工具逻辑。

### `scripts/env_check.py`

这是环境自检脚本，负责：

- 检查 Python、Torch、CUDA、OpenCV、Ultralytics 是否正常
- 检查默认模型能不能加载
- 打印模型类别名

它的作用是：**先确认环境没坏，再谈实时检测**。

### `scripts/camera_probe.py`

这是摄像头探测脚本，负责：

- 扫描不同摄像头索引
- 尝试不同 OpenCV 后端
- 检查哪个索引真的能读到帧

它的作用是：**先确认摄像头链路没问题，再谈模型推理**。

### `scripts/run_webcam.py`

这是 YOLO 实时检测主脚本，负责：

- 读取配置
- 打开摄像头
- 加载 YOLO 模型
- 对每一帧做推理
- 在画面上画框
- 显示和保存结果

## 5. YOLO 实时检测的核心原理

这个项目里最核心的脚本是 `run_webcam.py`。

它的工作原理可以拆成 6 个步骤。

### 第一步：读取配置

程序会把三个来源的配置合并成最终运行参数：

1. 代码里的默认值
2. `configs/default.yaml`
3. 命令行参数

优先级是：

**命令行参数 > YAML 配置 > 代码默认值**

这意味着：

- 你可以不改代码，直接改配置文件
- 也可以临时用命令行覆盖某些值

例如：

```powershell
python scripts/run_webcam.py --source 0 --backend dshow --save
```

这条命令只会覆盖：

- `source`
- `backend`
- `save`

其他参数继续沿用配置文件里的值。

### 第二步：加载模型

程序用 Ultralytics 的接口加载你训练好的权重：

```python
model = YOLO(str(model_path))
```

这一步的作用是把 `.pt` 权重文件变成一个可以直接调用的推理对象。

后面每一帧图像都可以这样送进去：

```python
results = model(frame, ...)
```

### 第三步：打开摄像头

程序底层用的是 OpenCV：

```python
cv2.VideoCapture(...)
```

这里会根据你选择的 `backend` 决定调用哪种 Windows 摄像头后端，比如：

- `dshow`
- `msmf`
- `auto`

同时还会尽量设置：

- 分辨率
- FPS
- FOURCC
- 缓冲区大小

这些参数不一定每个摄像头都完全遵守，但它们会直接影响：

- 实时帧率
- 延迟
- 兼容性

### 第四步：逐帧推理

程序进入一个 `while True` 循环，不断做这件事：

1. 从摄像头读一帧
2. 把这一帧交给模型
3. 得到检测结果
4. 把检测结果画回图像
5. 显示在窗口上

也就是说，它不是“先攒很多帧再一起算”，而是每次只处理一帧。

### 第五步：画结果

YOLO 返回的不是直接可显示的窗口，而是一个结果对象。

项目里通过：

```python
annotated = result.plot()
```

把检测框、类别名、置信度重新画回图像，得到“带标注的画面”。

然后又额外叠加了一个状态栏，显示：

- 当前帧号
- 当前 FPS
- 当前检测数量
- 当前模型文件名

这一步的意义是：你不需要看终端，也能知道程序是不是在持续工作。

### 第六步：保存结果

如果你加了：

```powershell
--save
```

程序会把处理后的画面持续写到一个视频文件里。

如果你加了：

```powershell
--save-txt
```

程序还会给每一帧输出一个标签文件。

所以这个项目不仅能“看”，还能“留档”。

## 6. 当前 YOLO 路线为什么可能检测不出来

你之前遇到“画面在动，但没有框”的情况，这从原理上通常不是程序没工作，而是下面几种原因：

### 1. 模型任务和当前画面不匹配

当前 YOLO 模型是缺陷检测模型，它只认识训练时学过的类别。

如果你拿它去看：

- 电脑屏幕
- 普通物体
- 大场景

它很可能输出：

```text
no detections
```

因为这些东西根本不在它的训练目标里。

### 2. 目标在图像里太小

如果裂缝或焊点只占图像很小一部分，模型可能根本学不到足够稳定的特征。

这就是你怀疑“裂缝太小”的原因，也是很合理的怀疑。

### 3. 输入分布和训练分布不一致

比如：

- 拍摄角度不同
- 光照不同
- 对着屏幕二次拍摄
- 缩放比例不同

都会让检测效果显著下降。

---

## 7. 这个项目当前为什么只保留 YOLO 路线

当前项目围绕的是“自动检测”这个核心目标。

YOLO 适合这条路线，因为它的职责就是：

- 自动在画面里找目标
- 告诉你目标大概在哪里
- 输出框和类别

这也和当前的主脚本、配置文件、日志结构保持一致，项目整体会更简单，后续维护和部署也更直接。

---

## 8. 为什么这个项目对后续单片机部署仍然有意义

虽然你现在运行的是 PC 版 Python 工程，但它对后续嵌入式部署仍然有很强参考价值。

因为真正需要验证的不是“这个窗口长什么样”，而是下面这条算法链路：

```text
输入图像
  → 预处理
  → 模型推理
  → 后处理
  → 输出结果
```

这条链路在 PC 端先跑通，有几个好处：

### 1. 先验证模型本身是否有用

如果 PC 上都检测不出来，后面搬到单片机也不会突然变好。

### 2. 先验证输入分布问题

你可以先知道：

- 相机角度是否合理
- 目标大小是否合理
- 现场光照是否合理

### 3. 先验证速度上限

PC 端可以帮助你估计：

- 输入尺寸多大比较合理
- FP16 有没有帮助
- 哪些后处理步骤最耗时

这些都会影响后续嵌入式版本的设计。

### 4. 先把“问题归因”分开

PC 版可以先回答：

- 是模型本身不行
- 还是输入图像不行
- 还是部署平台太弱

这个区分对后面做单片机实验非常重要。

## 9. 当前项目的优点和限制

### 优点

- 已经形成完整推理链路
- 能接真实摄像头
- 能实时显示结果
- 能保存视频和标签
- 能比较方便地改输入参数做对照实验

### 限制

- 当前 `run_webcam.py` 主要面向摄像头，不直接面向视频文件检测
- YOLO 路线依赖你训练好的模型是否适合当前场景
- 当前还是 PC 端验证，不是嵌入式端最终形态

---

## 10. 一句话总结这个项目的原理

如果只用一句话概括：

**这个项目就是把实时视频流接到模型推理链路上，让模型对每一帧做判断，再把结果重新画回画面并保存出来。**

其中：

- `YOLO` 路线解决“自动找目标”

你后面无论是继续改进检测，还是准备部署到单片机，本质上都还是在优化这条链路：

```text
图像输入 -> 模型推理 -> 结果输出
```
